# 04. GC와 힙 튜닝

## 힙이 크면 무조건 좋은가?

아니다. 핵심은 "컨테이너 한도 내에서 Non-Heap까지 고려한 적정값".

### 힙이 클 때 (MaxRAMPercentage=75%, 2048MB 컨테이너)

```
힙:        1536MB
메타스페이스:  ~100MB
코드캐시:     ~80MB
스레드스택:   ~200MB (200개 × 1MB)
NIO 버퍼:    ~50MB
합계:       ~1966MB  ← 2048MB에 바짝 붙음
여유:       ~82MB    ← 스파이크 오면 컨테이너 OOM Kill
```

### 힙이 적당할 때 (MaxRAMPercentage=50%, 2048MB 컨테이너)

```
힙:        1024MB
메타스페이스:  ~100MB
코드캐시:     ~80MB
스레드스택:   ~200MB
NIO 버퍼:    ~50MB
합계:       ~1454MB
여유:       ~594MB   ← 안전
```

## OOM은 2종류

### JVM OOM (java.lang.OutOfMemoryError)
- 힙 안에서 객체가 힙 한도를 초과할 때
- 힙이 크면 덜 나고, 작으면 더 날 수 있음
- 하지만 실사용량 700MB에 힙 1024MB면 40% 여유로 충분

### 컨테이너 OOM Kill (커널이 프로세스를 죽임) ← 진짜 위험한 것
- JVM 전체 메모리(힙 + Non-Heap)가 컨테이너 한도를 넘으면 발생
- 로그도 안 남고 태스크가 그냥 사라짐
- 힙을 75%로 잡으면 Non-Heap까지 합쳤을 때 한도에 거의 닿아서 위험

## GC: 자주 도는 게 나쁜 건 아니다

```
힙 1536MB (75%):
  → GC 거의 안 돌아도 됨
  → 대신 한번 돌면 1536MB 전체를 스캔 → GC 1회에 50~200ms (STW)

힙 1024MB (50%):
  → GC가 더 자주 돔
  → 대신 1024MB만 스캔 → GC 1회에 10~50ms (짧음)
```

G1GC는 힙이 클수록 GC pause가 길어진다.
사용자가 체감하는 "느림"은 GC 횟수가 아니라 GC pause 길이.
1초에 10ms짜리 2번 > 200ms짜리 1번.

## setof-commerce 실제 튜닝

변경 전: ECS 1024MB, MaxRAMPercentage=75% → 메모리 사용률 75%+, 간헐적 느림
변경 후: ECS 2048MB, MaxRAMPercentage=50% → 힙 1024MB + 여유 594MB → 안정적
