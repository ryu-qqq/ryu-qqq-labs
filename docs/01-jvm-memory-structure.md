# 01. JVM 메모리 구조

## 전체 구조

```
┌─────────────────────── 컨테이너 한도 (예: 2048MB) ──────────────────────┐
│                                                                          │
│  ┌──────────────── JVM 힙 (MaxRAMPercentage로 제어) ──────────────────┐  │
│  │                                                                     │  │
│  │  Young Gen (Eden + Survivor)  │  Old Gen                           │  │
│  │  - 새로 생성된 객체            │  - 오래 살아남은 객체                │  │
│  │  - Minor GC 대상              │  - Major/Full GC 대상               │  │
│  │                                                                     │  │
│  └─────────────────────────────────────────────────────────────────────┘  │
│                                                                          │
│  ┌──────────────── Non-Heap ──────────────────────────────────────────┐  │
│  │                                                                     │  │
│  │  1. Metaspace         ~100MB                                       │  │
│  │  2. Code Cache        ~80MB                                        │  │
│  │  3. Thread Stacks     ~200MB (스레드 수 × 1MB)                     │  │
│  │  4. Direct Buffers    ~50MB                                        │  │
│  │                                                                     │  │
│  └─────────────────────────────────────────────────────────────────────┘  │
│                                                                          │
│  OS 여유 공간                                                             │
└──────────────────────────────────────────────────────────────────────────┘
```

## 1. Metaspace (~100MB)

클래스를 로딩하면 여기에 저장된다.

저장되는 것:
- 클래스 구조 정보 (필드 목록, 메서드 시그니처, 상속 관계, 어노테이션)
- 메서드의 바이트코드

Spring Boot가 메타스페이스를 많이 먹는 이유:
- CGLIB 프록시: @Transactional, @Cacheable 붙은 빈마다 프록시 클래스 동적 생성
- 리플렉션: Spring이 빈 주입할 때 리플렉션 메타데이터 생성
- JPA: 엔티티마다 하이버네이트가 바이트코드 조작한 프록시 클래스 생성

## 2. Code Cache (~80MB)

JIT 컴파일된 네이티브 코드가 저장되는 곳.

```
1회차:  바이트코드 → 인터프리터가 한 줄씩 실행 (느림)
...
10000회차: "이거 자주 쓰네?" → JIT 컴파일러가 네이티브 코드로 변환
10001회차: 네이티브 코드 직접 실행 (빠름) ← Code Cache에 저장
```

JIT 최적화 기법들:
- 인라이닝: 자주 호출되는 작은 메서드를 호출부에 직접 삽입
- 탈출 분석: 객체가 메서드 밖으로 안 나가면 힙 대신 스택에 할당
- 루프 언롤링: 짧은 루프를 펼쳐서 분기 예측 실패 줄임
- 데드 코드 제거: 안 쓰는 분기 제거

JVM 시작 직후 느린 이유가 이것. Code Cache가 비어있어서 전부 인터프리터로 실행.
트래픽 받으면서 JIT가 점점 컴파일 → 이것이 "웜업".

## 3. Thread Stacks (~200MB)

각 스레드의 콜 스택. 스레드당 기본 1MB.

```
Thread-42 스택 (1MB):
┌──────────────────────────────────────┐
│ DispatcherServlet.doDispatch         │ ← 맨 아래
│ Controller.getProductGroups          │
│ Service.execute                      │
│ Repository.fetchThumbnails           │
│ QueryDSL.fetch                       │
│ HikariCP.getConnection               │
│ MySQL Driver.executeQuery            │
│         ... (빈 공간) ...            │ ← 1MB까지
└──────────────────────────────────────┘
```

스레드 200개 × 1MB = 200MB가 Non-Heap으로 잡힌다.
StackOverflowError는 이 1MB를 다 썼을 때 발생 (재귀 호출이 너무 깊어서).
